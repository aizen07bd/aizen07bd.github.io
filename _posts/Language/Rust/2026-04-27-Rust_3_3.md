Rust의 열거형(Enum)과 패턴 매칭(match)은 Rust를 '현대적이고 안전한 언어'로 만들어주는 가장 강력한 콤비입니다. 다른 언어의 Enum이 단순히 '숫자에 이름을 붙인 것'이라면, Rust의 Enum은 '데이터를 담을 수 있는 상태들의 집합'이에요.

------------------------------
## 1. 열거형(Enum) 정의: "이것 아니면 저것"
열거형은 여러 선택지 중 하나를 나타낼 때 사용합니다.
```rust
enum Message {
    Quit,                       // 데이터를 담지 않음
    Move { x: i32, y: i32 },    // 구조체처럼 이름을 가진 데이터를 담음
    Write(String),              // 튜플처럼 데이터를 담음
    ChangeColor(i32, i32, i32), // 여러 개의 데이터를 담음
}
```

* 특징: 위 4가지는 모두 Message라는 동일한 타입입니다. 구조체였다면 4개의 서로 다른 타입을 만들어야 했겠지만, 열거형은 하나로 묶어줍니다.

------------------------------
## 2. 패턴 매칭(match)과 함께 사용하기
열거형의 진가는 match를 만났을 때 발휘됩니다. 열거형 안에 든 데이터를 꺼내서 바로 사용할 수 있거든요.
```rust
fn process_message(msg: Message) {
    match msg {
        Message::Quit => {
            println!("종료합니다.");
        }
        Message::Move { x, y } => {
            println!("x: {}, y: {}로 이동!", x, y);
        }
        Message::Write(text) => {
            println!("메시지 내용: {}", text);
        }
        Message::ChangeColor(r, g, b) => {
            println!("색상 변경: R{}, G{}, B{}", r, g, b);
        }
    }
}
```

------------------------------
## 3. 왜 이렇게 쓰나요?

   1. 데이터 바인딩: match 구문 안에서 x, y, text 같은 변수 이름을 써주면, 열거형 안에 들어있던 값을 즉시 꺼내서(Binding) 사용할 수 있습니다.
   2. 안전성 (Exhaustiveness): 만약 Message에 새로운 종류를 추가했는데 match문에서 처리를 안 해줬다면? 컴파일러가 즉시 에러를 냅니다. "하나 빼먹었어!"라고 알려주는 거죠.
   3. 유연성: 서로 다른 모양의 데이터를 Message라는 하나의 타입으로 주고받을 수 있어 함수 설계를 단순하게 만듭니다.

------------------------------
## 4. 간결한 매칭: if let
만약 여러 상태 중 딱 한 가지 상태에만 관심이 있다면 match는 너무 길어질 수 있습니다. 이때 if let을 씁니다.
```rust
let some_msg = Message::Write(String::from("Hello"));
// "만약 some_msg가 Message::Write라면 그 안의 값을 text에 담고 실행해라"if let Message::Write(text) = some_msg {
    println!("메시지: {}", text);
}
```

* 강조: "Rust의 Enum은 단순한 상수가 아니라 데이터 타입을 하나로 묶어주는 그릇이다"라는 점을 꼭 언급해 보세요.


------------------------------
## 5. Option<T> : "값이 있을 수도, 없을 수도 있다"
Rust에는 다른 언어의 Null이나 None이 없습니다. 대신 Option 열거형을 사용합니다.

* 구조:
    * Some(T): 데이터(T)가 들어있는 상태
    * None: 값이 없는 상태

```rust
fn main() {
    let x: Option<i32> = Some(5);
    let y: Option<i32> = None;

    match x {
        Some(value) => println!("값이 있어요: {}", value),
        None => println!("값이 없네요."),
    }
}
```

* 왜 쓰나요? 값이 없는 변수를 실수로 사용해서 프로그램이 터지는 걸(Null Pointer Exception) 컴파일 시점에 원천 봉쇄합니다. Option에서 값을 꺼내려면 반드시 match나 if let으로 검사를 해야만 하거든요.

------------------------------
## 6. Result<T, E> : "성공하거나, 실패하거나"
작업이 성공했을 때의 결과값과 실패했을 때의 에러 정보를 함께 담는 타입입니다.

* 구조:
    * Ok(T): 작업 성공! 결과값 T를 담음
    * Err(E): 작업 실패! 에러 정보 E를 담음
    
```rust
fn divide(numerator: f64, denominator: f64) -> Result<f64, String> {
    if denominator == 0.0 {
        Err(String::from("0으로 나눌 수 없습니다.")) // 실패 상황
    } else {
        Ok(numerator / denominator) // 성공 상황
    }
}
fn main() {
    let result = divide(10.0, 2.0);

    match result {
        Ok(val) => println!("결과: {}", val),
        Err(e) => println!("에러 발생: {}", e),
    }
}
```

------------------------------
## 7. 더 간결하게 처리하기: unwrap()과 ? 연산자
매번 match를 쓰는 게 번거로울 때 사용하는 실전 팁입니다.

* unwrap(): "값이 무조건 있다고 확신해! 만약 없으면 그냥 프로그램 종료해(Panic)." (테스트용으로만 주로 씁니다.)
* ? (물음표 연산자): 에러가 발생하면 에러를 즉시 반환하고, 성공하면 값을 꺼내줍니다. 실무에서 가장 많이 쓰는 방식이에요.

------------------------------

* 제목: "Rust에는 Null이 없다? Option과 Result 완벽 가이드"
* 비유:
    * Option: "선물 상자". 열어보기 전까지는 비어있는지(None), 멋진 장난감이 들어있는지(Some) 알 수 없고, 반드시 열어보는 절차(match)가 필요함.
    * Result: "성적표". 시험 결과(Ok)가 들어있거나, 시험을 못 본 사유(Err)가 들어있음.
* 강조: "Rust는 에러를 '예외'로 던져서 무시하게 두지 않고, '반환값'으로 돌려주어 반드시 처리하게 강제한다"는 점을 언급해 주세요.


## 8. 이런것이 궁금해요.
학습과정에서 궁금한걸 미리 물어보고 답을 구했어요. 함께 확인해봐요.

### 1) 왜 그냥 if를 못 쓰고 if let을 쓰나요?
내용이 정확히 if let 문법이 탄생한 이유이자 핵심입니다.
말씀하신 대로 if는 기본적으로 bool 타입만 받을 수 있지만, Rust는 "패턴 매칭 성공 여부"를 조건처럼 쓸 수 있게 if let이라는 특별한 문법 설탕(Syntactic Sugar)을 제공합니다. 구체적으로 분석해 드릴게요.
일반적인 if 문은 값을 비교(Comparison)하지만, 열거형의 요소를 확인하는 과정은 패턴 매칭(Matching)이기 때문입니다.

* some_msg == Message::Write : ❌ (Enum 내부에 데이터가 들어있으면 직접 비교가 불가능함)
* if let 패턴 = 데이터 : ✅ "오른쪽 데이터가 왼쪽 패턴과 일치하느냐?"를 묻는 것입니다. 일치하면 true처럼 작동하여 블록 내부를 실행하고, 아니면 false처럼 건너뜁니다.

### 2) let 키워드가 들어가는 결정적인 이유 (바인딩)
질문하신 "일부러 let을 사용한 건가?"에 대한 답은 YES입니다. 단순히 '맞나 틀리나'만 확인하는 게 아니라, 맞다면 그 안의 데이터를 변수에 담아야(let) 하기 때문입니다.
```rust
// 1. match로 길게 쓸 때match some_msg {
    Message::Write(text) => println!("{}", text),
    _ => (), // 관심 없는 나머지는 무시
}
// 2. if let으로 간결하게 쓸 때if let Message::Write(text) = some_msg {
    // 1) some_msg가 Message::Write인지 확인하고 (조건)
    // 2) 맞다면 내부의 값을 꺼내서 'text'라는 변수에 담아라 (let 바인딩)
    println!("{}", text);
}
```

### 3) 블로그 한 줄 정리

"if let은 '패턴 확인'과 '변수 선언(let)'을 한 줄로 합친 문법입니다. bool 조건문처럼 작동하면서 동시에 내부의 데이터를 안전하게 꺼내 쓸 수 있게 해주죠."

### 4) 실전 팁

* 만약 안에 든 데이터가 궁금하지 않고 "이 타입이 맞는지"만 확인하고 싶다면 변수 자리에 언더바(_)를 쓰거나, matches! 매크로를 쓸 수도 있습니다.
```rust
if let Message::Write(_) = some_msg { ... } // 또는
if matches!(some_msg, Message::Write(_)) { ... }
```

이제 if let이 단순한 비교가 아니라 '조건부 변수 선언'이라는 점이 명확해지셨나요?

### 5) _ 패턴 (와일드카드)
모든 경우를 일일이 적기 힘들거나, 특정 몇 가지만 처리하고 나머지는 뭉뚱그려 처리하고 싶을 때 사용하는 방법이 당연히 있습니다.
Rust에서는 _ (언더스코어)가 바로 그 '나머지 전부'를 담당하는 키워드입니다.
match문의 마지막에 _를 사용하면, 앞에서 명시하지 않은 모든 나머지 경우를 처리합니다. 다른 언어의 switch문에 있는 default와 정확히 같은 역할을 합니다.
```rust
let msg = Message::Move { x: 10, y: 20 };
match msg {
    Message::Quit => println!("종료합니다."),
    _ => println!("나머지 메시지입니다."), // Quit을 제외한 Move, Write 등 모든 경우
}
```

### 6) 아무것도 하고 싶지 않을 때: ()
만약 나머지 경우에는 아예 아무런 코드도 실행하고 싶지 않다면, 빈 튜플인 ()을 사용합니다.
```rust
match msg {
    Message::Write(text) => println!("글쓰기: {}", text),
    _ => (), // "나머지는 아무것도 안 해"라는 뜻
}
```

### 7) 나머지 값을 변수에 담아 쓰고 싶을 때: other
나머지 전부를 처리하긴 하는데, 그 값이 무엇인지 나중에 써야 한다면 _ 대신 임의의 변수명(관례적으로 other나 unused)을 쓸 수 있습니다.
```rust
let dice_roll = 9;match dice_roll {
    3 => println!("3이 나왔네요!"),
    7 => println!("행운의 7!"),
    other => println!("{}번이 나왔으니 다시 던지세요.", other), // 나머지 숫자를 other에 담음
}
```

------------------------------
### 📝 블로그 핵심 포인트 (주의사항!)

* 순서가 중요합니다! _ 패턴은 반드시 맨 마지막에 와야 합니다. 만약 처음에 _ => ...를 써버리면, 컴파일러가 "어차피 여기서 다 걸러지는데 밑에 코드가 왜 있어?"라며 에러를 냅니다.
* 철저함(Exhaustiveness) 유지: _를 쓰는 순간 "모든 경우를 다루어야 한다"는 Rust의 규칙은 여전히 지켜지는 셈입니다. 컴파일러 입장에서는 "나머지 처리도 되어 있으니 안전하군!"이라고 판단하는 거죠.


### 8) Result<f64, String>의 진짜 의미
이 부분은 Rust의 제네릭(Generics)이라는 개념이 섞여 있어서 처음 보면 헷갈릴 수 있습니다. 
여기서 f64와 String은 "둘 다 사용 가능하다"는 뜻이 아니라, 각각의 역할이 고정되어 있는 것입니다.

* 첫 번째 자리 (f64): 작업이 성공(Ok)했을 때 담아올 데이터의 타입입니다.
* 두 번째 자리 (String): 작업이 실패(Err)했을 때 담아올 에러 정보의 타입입니다.

즉, "성공하면 f64를 주고, 실패하면 String으로 에러 메시지를 줄게!"라고 함수가 약속하는 것이죠.

### 9) Result(f64, String) (튜플 형태)와의 차이
질문하신 것처럼 만약 (f64, String)이라는 튜플을 반환한다면, 그것은 성공과 실패 데이터 두 개를 '동시에' 무조건 다 보낸다는 뜻입니다.

* 튜플 (f64, String): "여기 결과 숫자랑 에러 메시지 둘 다 담아서 보낸다. (에러가 없어도 빈 메시지라도 담아야 함)"
* 열거형 Result<f64, String>: "상자를 열어봐. 성공했으면 숫자(Ok)만 들어있을 거고, 실패했으면 에러 메시지(Err)만 들어있을 거야. 동시에 두 개가 들어있을 순 없어!"

### 10) 블로그 비교 요약

| 구분 | (f64, String) (튜플) | Result<f64, String> (열거형) |
|---|---|---|
| 의미 | 숫자와 문자열의 결합 | 성공 또는 실패의 선택 |
| 강제성 | 성공해도 에러 자리를 채워야 함 | 에러가 나면 성공 데이터는 신경 안 써도 됨 |
| 안전성 | 에러가 났는지 일일이 체크해야 함 | match 등을 통해 안전하게 분리 처리 강제 |

### 11) 약속된 위치 (Fixed Positions)
순서는 절대적으로 중요합니다! Rust의 Result<T, E> 타입은 이미 표준 라이브러리에 순서가 약속되어 정의되어 있기 때문입니다.
Result<T, E>에서 꺽쇠 안의 순서는 다음과 같이 고정되어 있습니다.

* 첫 번째 자리 (T): 성공(Ok)했을 때의 타입
* 두 번째 자리 (E): 실패(Err)했을 때의 타입

만약 Result<String, f64>라고 적는다면, 컴파일러는 "성공하면 문자열을 주고, 실패하면 에러 코드로 실수를 주겠구나"라고 반대로 해석합니다.

### 12) 왜 순서가 상관있을까요?
Ok와 Err는 타입이 명확해 보이지만, 실제로는 제네릭(Generics)이라는 틀을 사용하기 때문입니다.
```rust
// Rust 내부의 실제 정의 모습 (대략적)
enum Result<T, E> {
    Ok(T),  // 첫 번째 타입(T)은 Ok가 사용
    Err(E), // 두 번째 타입(E)은 Err가 사용
}
```

우리가 함수를 선언할 때 타입을 적는 행위는 이 틀에 "첫 번째 칸엔 이걸 넣고, 두 번째 칸엔 저걸 넣어줘"라고 지정하는 것과 같습니다. 따라서 순서를 바꾸면 데이터가 들어가는 '칸'이 바뀌어 버립니다.

### 13) 예시로 비교하기

* Result<i32, String>: "성공하면 숫자(i32), 실패하면 메시지(String)"
* Result<String, i32>: "성공하면 메시지(String), 실패하면 에러코드(i32)"

### 14) unwrap() : "값 없으면 프로그램 터트려줘"
unwrap()과 ? 연산자는 Option이나 Result 안에 담긴 값을 꺼내는 방법입니다. 
unwrap은 "안에 값이 무조건 있을 거야! 만약 없으면 나도 모르겠다!" 하고 배째라는 식의 방법입니다.
```rust
fn main() {
    let some_value = Some("사과");
    let none_value: Option<&str> = None;

    // 1. 값이 있는 경우: 안전하게 "사과"를 꺼냅니다.
    let fruit = some_value.unwrap();
    println!("내가 꺼낸 것: {}", fruit);

    // 2. 값이 없는 경우: 여기서 프로그램이 에러를 내며 강제 종료(Panic)됩니다.
    // let oops = none_value.unwrap(); 
}
```

* 언제 쓰나요? 주로 예제 코드를 짤 때나, 절대 값이 없을 리 없다고 확신할 때(테스트용) 사용합니다. 실제 서비스용 코드에서는 위험해서 잘 안 써요.

### 15) ? 연산자 : "성공하면 값 주고, 실패하면 에러를 밖으로 던져줘"
실무에서 가장 많이 쓰는 '에러 전파(Error Propagation)' 방식입니다. match문을 아주 짧게 줄여줍니다.
```rust
fn get_username() -> Result<String, String> {
    // 실제로는 파일이나 DB에서 읽어오겠지만, 여기선 예시를 위해 바로 생성
    Ok(String::from("RustUser"))
}
fn welcome_message() -> Result<String, String> {
    // get_username() 뒤에 붙은 '?'가 핵심입니다.
    // 1. 성공(Ok)하면 그 안의 "RustUser"를 바로 변수에 담아줍니다.
    // 2. 실패(Err)하면 이 함수(welcome_message)를 즉시 종료하고 에러를 밖으로 보냅니다.
    let name = get_username()?; 

    Ok(format!("어서오세요, {}님!", name))
}
fn main() {
    match welcome_message() {
        Ok(msg) => println!("{}", msg),
        Err(e) => println!("에러 발생: {}", e),
    }
}
```

### 16) match vs ? 비교
? 연산자가 얼마나 코드를 줄여주는지 비교해 보세요.

| 방식 | 코드 형태 | 특징 |
|---|---|---|
| match | match res { Ok(v) => v, Err(e) => return Err(e) } | 명확하지만 코드가 길어짐 |
| ? 연산자 | res? | 딱 한 글자로 에러 처리와 값 추출 끝! |


### 17) ?의 숨겨진 동작 원리
match는 Err(e)처럼 에러를 직접 변수에 담아서 다루는 게 눈에 보이는데, ?는 도대체 그 에러가 어디로 가는지 궁금하시죠?
결론부터 말씀드리면, ?는 에러를 '직접 처리'하는 도구가 아니라, '나를 호출한 놈에게 떠넘기는' 도구입니다. 
let name = get_username()?; 코드를 컴파일러는 내부적으로 아래와 같이 해석합니다.
```rust
// 우리가 쓴 코드let name = get_username()?;
// 컴파일러가 실제로 실행하는 코드 (의사 코드)let name = match get_username() {
    Ok(v) => v,                // 성공하면 값을 변수에 쏙!
    Err(e) => return Err(e),   // 에러나면 이 함수를 당장 끝내고 에러(e)를 밖으로 던짐!
};
```

즉, 에러 코드를 확인하고 처리하는 일은 ?를 쓴 함수가 아니라, 그 함수를 호출한 '상위 함수'가 맡게 됩니다.


### 18) 그러면 에러는 어디서 확인하나요?
결국 누군가는 match를 써서 에러를 확인해야 합니다. 보통은 프로그램의 입구인 main 함수나, 에러를 최종적으로 해결해야 하는 지점에서 확인합니다.
```rust
fn main() {
    // welcome_message 안에서 '?'로 던져진 에러가 결국 여기까지 흘러옵니다.
    match welcome_message() {
        Ok(msg) => println!("{}", msg),
        Err(e) => println!("여기서 에러 코드를 확인합니다: {}", e), // 👈 여기서 확인!
    }
}
```

### 19) ?를 쓰기 위한 대원칙 
이걸 모르면 ?를 쓸 때 컴파일 에러로 고생하게 됩니다.

* 반환 타입이 일치해야 함: ?가 에러를 return 하기 때문에, ?를 사용하는 함수의 반환 타입도 반드시 Result나 Option이어야 합니다.
* 에러 타입의 변환: 가끔 String 에러를 던졌는데 받는 쪽은 io::Error를 원할 때가 있습니다. 이때 ?는 From 트레이트라는 것을 이용해 자동으로 에러 타입을 변환해 주기도 하는 아주 똑똑한 녀석입니다.

### 20) "예외(Exception)"가 없기 때문입니다
"실패할 가능성이 조금이라도 있는" 함수나 메소드는 거의 예외 없이 Result를 반환하는 것이 Rust의 표준 문법이자 철학입니다.
Java나 Python 같은 언어는 실행 도중 에러가 나면 throw를 통해 예외를 던집니다. 하지만 Rust는 예외를 던져서 흐름을 끊는 대신, 에러를 하나의 '데이터'로 취급해서 함수 결과값에 섞어서 돌려줍니다.

* 다른 언어: "일단 실행해, 에러 나면 딴 데서 잡겠지."
* Rust: "이 함수는 실패할 수도 있어. 결과 상자(Result)를 줄 테니 네가 직접 열어서 확인해."

### 21) 컴파일러가 처리를 강제하기 때문입니다
Result를 반환값으로 받으면, 개발자는 그 값을 사용하기 위해 반드시 match, if let, 또는 ?를 써야만 합니다.

* 만약 Result를 반환했는데 이를 무시하고 지나가려 하면, 컴파일러가 "사용되지 않은 Result가 있습니다(unused Result)"라는 경고를 띄워 실수를 방지합니다.

### 22) 실전 예시: 표준 라이브러리
우리가 앞으로 배울 4단계의 내용들도 대부분 Result를 씁니다.

* 파일 열기 (File::open): 파일이 없을 수도 있으므로 Result 반환.
* 네트워크 통신: 연결이 끊길 수 있으므로 Result 반환.
* 문자열을 숫자로 바꾸기 (parse): "abc"를 숫자로 바꿀 순 없으니 Result 반환.
